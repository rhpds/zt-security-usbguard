# Real World Examples of USBGuard

A BadUSB is a security exploit that turns a simple USB device—like a thumb drive, a charging cable, or even a specialized tool like a Flipper Zero—into a malicious tool by attacking the firmware rather than the stored data.

Unlike a traditional "infected" USB drive that contains a virus file (which an antivirus can scan), a BadUSB device tricks your computer into thinking it is something else entirely, most commonly a keyboard.

## How It Works: The HID Attack

The most common form of BadUSB is a Human Interface Device (HID) attack. Here is the breakdown of why it is so effective:

* **Impersonation:** When you plug the device in, it identifies itself to the operating system as a standard USB keyboard.

* **Inherent Trust:** Computers are designed to trust keyboard input implicitly. They don't ask for permission to let a keyboard "type."

* **The Payload:** The microcontroller inside the device executes a pre-programmed script. It can "type" at superhuman speeds (thousands of words per minute).

* **The Result:** Within seconds of being plugged in, the device can open a terminal, type out a command to download a backdoor, execute it, and close the window before the user even notices.

## Popular BadUSB Tools

While attackers can repurpose generic USB controllers, several well-known tools are built specifically for this purpose:

* **USB Rubber Ducky:** The "gold standard" for this attack, created by Hak5. It looks exactly like a standard flash drive but is a high-speed script injector.

* **Flipper Zero:** A multi-tool for pentesting that includes a "Bad USB" mode. It allows you to store and run various scripts from a digital interface.

* **Malicious Cables (e.g., O.MG Cable):** These look like standard Lightning or USB-C charging cables but contain a hidden Wi-Fi chip and microcontroller that allows a hacker to trigger keyboard commands remotely over the air.

## Why Antivirus Can't Stop It

Standard Antivirus (AV) software looks for malicious code on a disk. However, a BadUSB device doesn't necessarily "store" a file on the computer. It simply "types" commands. To the computer, it looks like the user is just typing very quickly.

## How to Protect Yourself

* **Physical Security:** Never plug in a USB drive you found in a parking lot or received as a "free gift" from an untrusted source.

* **Software Whitelisting(Allow Listing):** Use tools like USBGuard (on Linux) or GPOs (on Windows) to block any USB device that isn't on a pre-approved list of Hardware IDs.

* **Hardware Blockers:** Use "USB Data Blockers" when charging in public ports; these physically disconnect the data pins and only allow power to flow.

You will have to verify what is plugged into your machine, and not just accecpt that what is plugged in is trusted.

* Logitech Keyboard via VID:PID which is 046d:c31c
* Flipper Zero VID:PID is 0483:5740
* Hak5 USB Rubber Ducky (standard) VID:PID is 1eaf:0030
* OMG Cable VID:PID is d3c0:d34d

Example output adding the blocking for these devices:
[source,bash,role="execute",subs=attributes+]
```
sudo tee /etc/usbguard/rules.conf > /dev/null <<'EOF'
allow id 046d:c52b serial "1234ABCD" name "Logitech USB Receiver"
allow id 046d:c31c
reject id 0483:5740
reject id 1eaf:0030
reject id d3c0:d34d
allow with-interface 09:00:00
allow with-interface 03:00:*
block with-interface 08:*:*
block
EOF
```

Before applying, validate syntax:

[source,bash,role="execute",subs=attributes+]
----
sudo usbguard read-descriptor /etc/usbguard/rules.conf
----

If no errors, the file is valid.

Then Restart USBGuard Service

Apply the new policy:

[source,bash,role="execute",subs=attributes+]
----
sudo systemctl restart usbguard.service
----

Check which devices are currently allowed:

[source,bash,role="execute",subs=attributes+]
----
sudo usbguard list-devices
----
